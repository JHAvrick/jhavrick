# Self-Hosting A Chrome Extension

Did you know it's possible to self-host a Chrome extension? Don't get too excited, there are *a lot* of caveats. 

## Contents
[Prerequisites](#prerequisites) \
[Dependencies](#dependencies) \
[Sign and Pack the Extension](#sign-and-pack-the-extension)

## Why would you want to do this?

 - You have developed an extension for use in an Enterprise organization and you don't want the extension published on the Chrome Web Store, even if unlisted
 - You want to deploy and test your extension in a production-like environment without first publishing to the Chrome Web Store

We are really only going to be focusing on the latter here, but these steps may still be enlightening for the first use case.

## Summary

We are going to use the [crx](https://www.npmjs.com/package/crx) module to sign and pack our extension and then create a barebones [express](https://expressjs.com/) server to host two endpoints - one for our `update.xml`, which specifies the current extension version and location and one to serve the packaged extension itself. We will generate a temporary public URL for our express server using [ngrok](https://www.npmjs.com/package/ngrok) and then force deploy the extension to our device(s) from the [Workspace Admin Console](https://admin.google.com/). We will put this all in one script called `deploy.js`.

## Prerequisites

 - [Google Workspace](https://admin.google.com/) (previously GSuite) Enterprise Account
 - A [device or browser enrolled in your organization](https://admin.google.com/ac/chrome/devices)

For the sake of brevity we're going to assume you already have a functioning Chrome Extension, Node, and the will to forge ahead.

## Dependencies

Here's what we're going to need as far as dependencies:
 - [crx](https://www.npmjs.com/package/crx) - We'll use this to automate packing our extension as a CRX. You can also do this step manually each time using Chome if you desire.
 - [express](https://www.npmjs.com/package/express) - We'll use express to host our extension and `update.xml` file.
 - [ngrok](https://www.npmjs.com/package/ngrok) - We'll use ngrok to temporarily host the extension. You may need to create a free account. If you already have a more permanent hosting solution in mind, you can skip this.
```js
npm i --save-dev crx ngrok express
```

## Step 1: Sign, Pack the Extension
---

Signing the extension will generate a `.pem` key file which can be used for future packings and is necessary to maintain the same extension ID each time. This ID will *not* be the same as your production extension ID unless your already have a production key file. I would not recommend using your production key file.

#### Using Chrome

If you're a glutton for punishment you can pack your extension each time using this method. You probably know the drill - go to `chrome://extensions` and click the "Pack extension" in the top left. 

![pack chrome extension](/chrome-pack-ext.png)

Specify the root directory of your extension and add your `.pem` key file if you have one. If you don't, this step will generate one for you. Keep track of that.


#### Using CRX

In the intereset of saving ourselves a few clicks, we can use `crx`. This will be the first step in our deployment script.









If you've never packed the extension, simply navigate to your extension's root directory and run the following command:

`
crx pack -o
`

This command will also generate a `.pem` key file for you or use your existing key file if it is in the root directory. If you want to use a specific key file, you can specify it with the `-p` flag:


`
crx pack myFirstExtension -p myPrivateKey.pem -o
`

 Using the same `.pem` file to sign your extension each time is necessary to maintain the same extension ID, so keep track of this file. If you lose your dev `.pem` it may not be the end of the world, but you will need to redeploy the self-hosted extension.

 These examples are taken directly from the [crx](https://www.npmjs.com/package/crx) NPM guide.
